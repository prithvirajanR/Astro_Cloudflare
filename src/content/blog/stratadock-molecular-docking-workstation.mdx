StrataDock was developed to make a common docking workflow more repeatable: receptor preparation, ligand handling, binding-site definition, docking execution, scoring, visualization, and report export. The project is best understood as workflow engineering for computational structural biology rather than as a claim that docking scores alone can solve binding prediction.

The practical use case is intentionally messy. A user may have a receptor PDB, a ligand file or small ligand library, no reference ligand, and no clean docking box. StrataDock provides a structured path from input files to ranked outputs.

![StrataDock run screen showing docking configuration and terminal progress](/blog/stratadock-run.jpg)

## Workflow scope

The workflow covers receptor cleanup, ligand preparation, binding-site definition, docking, scoring, visualization, and export. It supports AutoDock Vina by default and GNINA as an optional CPU/CUDA path.

Binding-site setup is deliberately flexible:

- reference ligand mode for redocking and known binding sites
- fpocket mode when only a receptor structure is available
- manual box mode when coordinates are already known

This is deliberately broader than a single demonstration dataset. Real docking work often begins with incomplete structural context, so the application separates the binding-site decision from the docking engine itself.

## Validation design

The repo includes five co-crystal redocking validation systems:

- HIV protease, 1HSG
- EGFR kinase, 1M17
- ABL kinase, 1IEP
- trypsin, 3PTB
- HSV thymidine kinase, 1KIM

Each validation case includes receptor files, native ligand files, ligand SDF inputs, docking box coordinates, and checks for pose files, docking scores, and pose-level RMSD thresholds.

The validation set is the most important engineering result. It turns StrataDock from a user-interface wrapper into a workflow with testable behavior and known reference cases.

## Implementation notes

The repo is split into core modules for docking, reports, pockets, receptors, ligands, sessions, and visualization. The Streamlit app is large, but the important logic lives in reusable modules under `src/stratadock/core`.

The test suite covers ADMET-style profiles, batch runs, binding boxes, diagnostics, GNINA handling, pocket prediction, receptor and ligand preparation, reports, sessions, UI tables, and validation data.

## Limitations and next steps

Docking output should be treated as triage, not a definitive binding claim. The validation cases help confirm that the pipeline behaves sensibly on known systems, but they do not establish broad predictive accuracy across all receptors, ligands, or scoring functions.

The next version should continue separating core workflow logic from the Streamlit surface, add more visual and command-line regression checks, and report interface-quality metrics more explicitly for validation cases.
